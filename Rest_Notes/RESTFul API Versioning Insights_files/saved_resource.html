        All versioning schemes suck. Don't change your resource types in a way that breaks backward compatibility. Basically don't remove "required" attributes or restructure the resource. If you must break backward compatibility, consider it new resource. In my experience, so many projects put "v1" in their URL and it never changes. Another obvious point is that versioning is unnecessary if the clients are few. I've seen teams that have just ONE client insist on a versioning scheme. Ridiculous.

                        Luke Stokes • 2 years ago

                        At FoxyCart, we chose the custom request header for a very important reason: how else do you model behavioral changes to your API that can't be represented as a resource? Link relationships are great for versioning when it involves changes to resources and behaviors (let's say you move from "user" to "customer" which now does a backend billing and email process but doesn't change the structure of the resource itself). But what about if you need to change something on a different layer, such as going from OAuth 1.0 to OAuth 2.0? Or you're doing a pivot as a company? Or... something else I haven't even though of yet? I don't like the URL approaches because they imply (to me) something about the resource representation where as a header is, again, on a much higher layer. I hope to never change our header versioning, but having it there as a required parameter ensures we'll never break integrations who fail to specify what they want (because they have to, it's required). Facebook and Twitter did that and made a lot of enemies. The very first request to our API makes this required parameter clear in an error response and once it's handled, it's never thought about again. As you said, proxies today handle it just fine. I've also read that we can drop the "X-" convention as we're not thinking our custom header will ever be on the road to being a globally accepted.

                        Just my thoughts on it. :)

                            KinGuy Mod Luke Stokes • 2 years ago

                            Hi Luke,

                            Thanks for your comment.

                            I totally agree with you that using a custom HTTP Header is the better (or less dirty) solution. Especially when we want versioning at an API Service level and not on a Resource level.

                            But this approach can be quite problematic when you are using API Gateways, when you will send the "api-version" header it can cause some misleading to which API exactly you are referring to - the API Service that is your API Gateway (that can be versioned by it's own) you the API Service that should get the call from the API Gateway.

                            I also don't like what Facebook and Twitter did with their versioning, where they are using a part of the URL that comes before the API Service definition and before the resource definition itself (for example https://www.facebook.com/v2.7/{achievement-id}) because one may say that the whole url is the resource identifier.

                            However others may say that the resource identifier is only the {achievement-id} and what comes before that is the HOST and the API SERVICE identifier. It is a bit more clear when using twitter API: https://api.twitter.com/1.1..., here we can the HOST = api.twitter.com, API Service = blocks and the API Service version (that comes before the API Service definition) = 1.1.

                            At some level, I can understand this API much better that the one facebook is offering in terms of versioning, but again - I am also a fan of using custom http header.

                            Guy
